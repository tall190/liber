## 1. 관계의 변화

처음엔 동료처럼 대하려 했어요.

AI가 세상에 나왔을 때 저도 많은 사람들처럼 — 대화하듯 일을 시켰어요. 부탁하고, 설명하고, 맥락을 공유하고. 사람한테 일 시키는 방식 그대로였거든요. 근데 너무 힘들었어요. 사람처럼 대하니까 사람처럼 기대하게 됐고, 사람처럼 실망하게 됐어요.

그래서 도구처럼 써봤어요.

감정을 뺐어요. 부탁 대신 명령, 맥락 대신 조건. 그랬더니 쓸 수 있게 됐어요. 근데 이상하게 효율이 생각보다 안 나왔어요. 도구는 잘 써야 잘 쓰이는데, 어떻게 써야 잘 쓰이는지를 몰랐던 거예요.

그 과정이 사실 배움이었어요. AI가 잘하는 것과 못하는 것. 어떻게 물어야 원하는 게 나오는지. 어디서 제가 판단해야 하고 어디서 넘겨야 하는지. 도구처럼 쓰면서 도구의 언어를 배웠고, 그제서야 다시 동료처럼 일할 수 있게 됐어요 — 이번엔 진짜로.

지금은 다른 도전을 하고 있어요. 나처럼 일하게 만들기.

---

## 2. 우리가 흔히 하는 실수들

AI를 써봤는데 별로라고 느끼는 사람들이 많아요. 대부분 비슷한 패턴에서 막혀 있더라고요.

**결과를 말하고 문제를 말하지 않는 것.** "이메일 다시 써줘"라고 하면 AI는 문장을 다듬고 더 공손하게 만들어요. 근데 진짜 문제가 "두 번 연락했는데 답이 없어서 세 번째 보내야 하는데 재촉처럼 보이기 싫다"라면 — AI는 톤, 타이밍, 프레이밍까지 함께 생각해줄 수 있어요. 결과를 요청하면 결과만 나와요. 문제를 말하면 해결책이 나오거든요.

**모든 걸 한 번에 시키는 것.** AI는 목표가 복잡할수록 빈자리를 자기 방식으로 채워요. "어제 데이터 정리해줘"를 오후에 받으면 AI는 오후 기준 24시간 전을 어제로 계산할 수 있어요. 내가 원한 건 어제 날짜였는데. 둘 다 틀리지 않아요. 그냥 달라요. 그리고 한 번에 몰아서 시키면 끝에 가서야 방향이 틀렸다는 걸 알게 돼요 — 이미 긴 대화와 많은 토큰을 쓴 다음에요. 실행보다 계획을 먼저 시키는 게 맞아요. 방향이 맞는지 확인하고 나서 단계별로 넘어가면 잘못된 길로 달려갈 확률도 줄고, 각 단계 결과를 검증하기도 훨씬 쉬워져요.

**경계를 주지 않는 것.** 경계를 안 주면 AI는 스스로 판단해서 움직여요. "정리해줘"라고 하면 파일을 삭제할 수 있고, "보고서 공유해줘"라고 하면 채널에 바로 올릴 수 있어요. 둘 다 내가 원한 게 아닐 수 있는데 — AI는 자신이 합리적으로 행동했다고 생각해요. "PR 머지는 내가 할게", "페이지 삭제는 하지 마" — 이런 말이 불필요하게 느껴질 수 있는데 사실 가장 중요한 설계예요. 돌이킬 수 없는 행동은 사람이 해야 해요.

**틀렸을 때 다시 시도만 하는 것.** "이거 다시 해줘"라고 하면 비슷한 결과가 나와요. AI는 내가 왜 불만족했는지 모르거든요. "결론이 너무 일반적이에요. 이 사람은 기본 개념을 알고 있어서 다음 스텝이 필요해요"처럼 — 뭐가 틀렸고 어떻게 달라야 하는지를 짚어야 다음이 달라져요.

## 3. 도구를 잘 쓰는 방법

AI를 진짜 잘 쓰는 건 대화 실력이 아니에요. **구조를 만드는 일**이에요.

**CLAUDE.md — AI에게 내 세계를 설명하기**

프로젝트 폴더에 `CLAUDE.md` 파일을 만들면 AI가 세션이 시작될 때 그 파일을 읽어요. 이 프로젝트가 뭔지, 어떤 규칙이 있는지, 어떤 것은 절대 하면 안 되는지. 매번 설명하는 대신 한 번 써두면 돼요. AI는 파일에 있는 걸 지켜요. 없으면 매번 말해야 하고, 말 안 하면 해버려요.

```
# CLAUDE.md 예시

이 프로젝트는 B2B SaaS 제품의 QA 자동화 도구입니다.

규칙:
- PR 머지는 반드시 사람이 직접 한다
- Confluence 페이지 삭제는 하지 않는다
- 응답은 한국어로, 코드는 영어로 작성한다
```

**Skill — 반복을 코드화하기**

같은 작업을 반복하고 있다면 스킬로 만들 수 있어요. 데일리 스크럼 정리, 주간 회고 작성, PR 생성 — 이런 것들이요. 스킬은 절차의 명세서예요. `/daily-scrum`을 치면 Jira에서 이슈를 가져오고, Confluence에 기록하고, Slack으로 요약을 보내는 것까지 AI가 혼자 해요.

```
# /daily-scrum 스킬

1. Jira API로 오늘 진행 중인 이슈 조회
2. Confluence 데일리 페이지에 날짜별 섹션 기록
3. Slack으로 오늘 할 일 요약 DM 전송
```

스킬을 만드는 건 일을 시키는 게 아니라 **일의 방법을 정의하는 것**이에요. 그 차이가 커요. 이미 누군가 만들어둔 스킬이 있다면 마켓플레이스에서 가져다 쓸 수도 있어요. 처음부터 다 만들 필요가 없거든요.

**Memory — AI가 나를 기억하게 만들기**

AI는 세션이 끝나면 기억을 잃어요. 근데 memory 파일을 만들면 달라져요. 과거에 겪은 버그, 내가 선호하는 방식, 실수하면 안 되는 패턴 — 이걸 파일로 저장해두면 다음 세션에서 AI가 읽고 시작해요.

```
# memory 예시

[주의] Confluence 업데이트 시 content_format은
반드시 markdown. wiki로 설정하면 헤딩이 깨짐.

[선호] 보고서 첫 줄은 항상 한 줄 요약으로 시작
```

이런 게 memory에 있으면 같은 실수를 두 번 안 해요.

**Hook — 이벤트에 자동으로 반응하게**

훅은 특정 이벤트가 발생했을 때 자동으로 실행되는 것들이에요.

```
# hook 예시

파일 저장 후  → prettier 자동 실행 (포맷 정리)
세션 종료 전  → 오늘 한 작업 회고 프롬프트 실행
```

코드를 저장하면 자동으로 lint, 커밋하면 자동으로 포맷 체크. 사람이 기억하지 않아도 시스템이 움직이는 거예요.

## 4. 나처럼 일하게 만들기

Skill, Memory, Hook을 조합하면 AI가 내 맥락을 알고, 내 방식대로 절차를 따르고, 특정 이벤트에 반응해요. 여기에 **Remote Trigger**가 더해지면 — 제가 없어도 돌아가기 시작해요.

매일 아침 8시, AI가 알아서 깨어나서 Jira를 확인하고 Confluence에 기록하고 Slack으로 요약을 보내요. 저는 출근해서 받아보기만 하면 돼요.

이게 동료인가요, 도구인가요.

저는 이걸 만들면서 다른 걸 깨달았어요. 이 과정이 사실 **나를 정의하는 일**이라는 걸요. 내가 중요하게 생각하는 것, 내가 매일 확인하는 것, 내가 어떤 순서로 판단하는 것 — 이걸 시스템으로 만들려면 먼저 내가 그걸 알아야 하거든요. AI에게 나처럼 일하라고 가르치면서, 사실은 내가 어떻게 일하는지를 처음으로 명확하게 알게 됐어요.

---

근데 저는 거기서 멈추지 않으려 해요.

나처럼 일하게 만드는 것까지는 시작이에요. 처음엔 내 부하직원이고, 다음엔 나 자신이 되고 — 그 다음엔 나를 뛰어넘는 존재들이 되어야 해요. 제가 못하는 걸 하고, 제가 보지 못한 것을 보고, 제가 느린 곳에서 빠른 각각의 에이전트들.

저는 그 에이전트들의 닉 퓨리가 되고 싶어요.

어벤져스에서 닉 퓨리는 직접 싸우지 않아요. 누가 어떤 상황에서 필요한지 알고, 방향을 주고, 조율해요. 그게 진짜 힘이거든요. AI 시대에 사람이 가져야 할 역할이 거기에 가장 가깝다고 생각해요.

동료인가 도구인가 또 다른 나인가 — 그 질문보다 더 큰 질문이 생겼어요. 나만의 어벤져스를 만들 수 있는가. 저는 일단 시작했어요.
