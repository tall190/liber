AI 코딩 어시스턴트를 쓰는 개발자가 빠르게 늘고 있다. Claude, Copilot, Cursor — 이름은 달라도 약속은 같다. "더 빠르게 개발할 수 있다."

실제로 빠르다. 보일러플레이트 코드는 몇 초 만에 나오고, 리팩토링 제안은 꽤 정확하다. 그런데 개발 속도가 빨라진 만큼, 테스트와 QA가 따라오고 있을까?

아니다. 오히려 병목이 생겼다.

---

## AI가 빠르면, QA가 느려진다

코드 생산 속도가 빨라지면 검증 속도도 같이 빨라져야 한다. 그렇지 않으면 병목은 QA로 옮겨진다. 개발자가 하루에 10개의 기능을 만들 수 있어도, QA가 3개밖에 검증하지 못하면 전체 속도는 3개 기준으로 수렴한다.

해결책은 QA도 AI로 자동화하는 것이다. E2E 테스트를 AI가 작성하고, 회귀 케이스를 AI가 추적하고, 릴리스 전 체크리스트를 AI가 실행한다.

그런데 여기서 문제가 생긴다.

**AI가 당신의 버튼을 못 찾는다.**

---

## AI는 UI를 어떻게 "보는가"

사람은 버튼을 보면 안다. 파란 색, 둥근 모서리, "저장" 이라는 텍스트. 클릭하면 된다.

AI는 다르다. AI가 UI를 다루는 방식은 크게 두 가지다.

**1. DOM 탐색**: HTML 구조를 파싱해서 요소를 식별한다.
**2. 시각 인식**: 스크린샷을 보고 좌표를 추정한다.

문제는 둘 다 깨지기 쉽다는 것이다. 디자인이 바뀌거나, 텍스트가 수정되거나, 컴포넌트 위치가 달라지면 AI는 다시 헤맨다.

반면, 이렇게 만든 버튼은 다르다:

```tsx
<button data-testid="save-draft-button">
  저장
</button>
```

`data-testid="save-draft-button"` — 이 속성 하나가 AI에게 명확한 계약을 제공한다. "이 버튼은 '임시저장' 버튼이다. 디자인이 바뀌어도, 텍스트가 바뀌어도, 이 이름으로 찾아라."

이것이 **AI-Readable UI**의 핵심이다.

---

## 구조화된 UI가 AI에게 주는 것

`data-testid`는 단순한 테스트 도구가 아니다. AI에게 **의미 있는 좌표계**를 제공한다.

일반적인 UI에서 AI가 "임시저장 버튼"을 찾으려면:
- 텍스트가 "저장"인 버튼 탐색 → "저장하기", "SAVE", "임시 저장" 등 변형에 취약
- CSS 클래스 탐색 → 리팩토링 한 번에 무너짐
- 시각 인식 → 레이아웃 변경에 취약

`data-testid`가 있으면:
- `[data-testid="save-draft-button"]` — 언제나 정확히 찾는다

네이밍 컨벤션만 팀 내에서 통일해도, AI가 테스트를 작성하는 속도와 정확도가 극적으로 올라간다. 우리 팀에서는 아래 패턴을 사용한다:

```
nav-{route}               # 네비게이션 링크
{entity}-list             # 리스트 컨테이너
{entity}-{index}-row      # 테이블 행
{action}-button           # 액션 버튼
{field}-input             # 입력 필드
{section}-section         # 영역 구분
```

패턴이 예측 가능할수록, AI의 추론 오류가 줄어든다.

---

## 이것은 QA 기법이 아니다

AI-Readable UI를 "E2E 테스트를 위한 것" 으로만 이해하면 놓치는 게 있다.

이건 **AI와 함께 개발하기 위한 인프라**다.

AI 코딩 어시스턴트에게 "이 기능에 대한 테스트 작성해줘" 라고 하면, AI는 어떻게 동작할까? 컴포넌트의 구조를 읽고, 어떤 요소가 어떤 역할을 하는지 추론해야 한다. UI에 시맨틱 구조가 있으면 AI의 추론이 정확해진다. 그렇지 않으면 AI는 코드를 읽어도 의도를 파악하지 못한다.

개발 단계에서 AI가 컴포넌트 구조를 빠르게 이해하고, 테스트 단계에서 AI가 정확하게 요소를 찾고, QA 단계에서 AI가 신뢰할 수 있는 검증을 수행하는 것 — 이 전 사이클이 **UI가 AI에게 읽히는 구조**일 때만 가능하다.

---

## 구체적으로 무엇을 해야 하는가

**1. `data-testid` 컨벤션 도입**

모든 인터랙티브 요소에 `data-testid`를 붙인다. 팀 내 네이밍 컨벤션을 문서화하고, 리뷰 체크리스트에 포함한다.

**2. 시맨틱 HTML 사용**

`<div onClick>` 대신 `<button>`, `<nav>`, `<main>`, `<section>`. 의미 있는 태그는 AI가 구조를 추론하는 데 직접적으로 도움을 준다.

**3. ARIA 레이블 작성**

`aria-label`, `aria-describedby` — 스크린리더를 위한 것이기도 하지만, AI의 맥락 파악에도 그대로 활용된다.

**4. 컴포넌트 경계를 명확히**

컴포넌트 단위로 `data-testid`를 부여해 AI가 "이 영역이 어떤 기능을 담당한다"를 인식하게 한다.

---

## 결국 사람도 이익이다

AI-Readable UI를 만드는 것은 AI를 위한 것만이 아니다.

명확한 `data-testid`는 사람인 개발자가 테스트를 작성할 때도 편하다. 시맨틱 HTML은 접근성을 높인다. ARIA 레이블은 다국어 확장 시에도 일관성을 유지한다.

AI가 읽기 좋은 UI는 사람도 읽기 좋은 UI다.

---

개발 속도가 AI 덕분에 빨라졌다면, 다음 병목은 검증이다. 검증까지 AI와 함께 하려면, UI부터 AI가 읽을 수 있어야 한다.

당신의 버튼에 이름을 붙여라. AI가 찾을 수 있도록.
